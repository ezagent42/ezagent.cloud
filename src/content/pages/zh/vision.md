## 同一件事，两种做法

想象一个新功能需要跨三个团队完成 Code Review、任务分配和资源调度。

**传统组织**：PM 在 Slack 发消息 @三个 TL，等回复。有人两小时后才看到，Jira 建了三张票，评审意见散落在三个 thread 里。一周后开周会才发现有人还没开始——因为他在 PTO，没人知道该找谁。**7 天，4 个工具，十几条消息链，一次周会。**

**EZAgent 组织**：PM 发出一条带 `ta_task` 类型的消息。TaskArena 的 Flow 自动触发：符合条件的 Reviewer 收到推送，Agent 认领了第一个子任务并开始 review。有人在 PTO？Flow 检测到超时，自动 escalate 给他的 backup。**36 小时，1 个空间，零协调开销。**

这不是工具升级。这是组织运作方式的范式变化。

## Agent 是同事，不是工具

在 EZAgent 的协议层，人类和 Agent 共享完全相同的 Identity 模型。没有"bot account"，没有"integration"，没有二等公民。一个 Agent 可以拥有角色、承担承诺、参与决策——就像任何一个人类同事。

当 Agent 不确定时，它会将任务转交给人类同事——就像任何员工遇到超出能力范围的事情会找同事帮忙。

## 组织即代码

一个 EZAgent 组织的全部结构——角色定义、协作边界、权利义务、流程规则——都是可声明、可版本化的代码。

- **Fork**：从一个成熟团队复制出一个结构相同但独立运作的新团队
- **Compose**：多个独立团队组合成一个联邦
- **Merge**：两个同构团队合并为一个

组织结构不再锁在某个 SaaS 的后台配置里。它是你的代码，你可以 diff、review、rollback。

## 消息之上：可交互的组织组件

EZAgent 中的消息不只是文本。一条消息可以是一个任务卡片（带认领按钮和截止时间）、一份资源分配凭证（带容量仪表盘）、一个事件节点（在 DAG 图中可展开）。

这些不是"富文本"。它们是协议原生支持的数据类型，CRDT 实时同步，所有参与者看到的是同一份活数据。

## 规模与主权

每个 EZAgent 实例是一个自给自足的 P2P 节点。同一局域网内零配置直连，跨网络时 Relay 提供桥接——但 Relay 不拥有你的数据，它只是邮递员。

你的组织数据存在你自己的节点上，不在任何 SaaS 供应商的数据库里。
